1. Document Purpose 
This document defines the complete visual system and frontend interaction rules for 
ParkingSol. Its purpose is to eliminate design ambiguity before implementation starts so 
the frontend can be built consistently, beautifully, and fast. This is not a mood board. 
This is the canonical UI contract for the MVP. 
ParkingSol is an operational parking intelligence platform. Its interface must feel 
premium, modern, and memorable, but it must also remain readable under pressure. 
Users are not opening the app to admire visuals alone. They are opening it to answer 
urgent operational questions: 
• What just happened? 
• Is the vehicle paid or unpaid? 
• Is there an active violation? 
• Does this require action now? 
• Is the system healthy? 
• What changed? 
• Who needs to be notified? 
That means the frontend must balance two things at once: 
1. A distinct visual identity that feels premium, polished, and impressive. 
2. A decision-first workflow where operators can read and act instantly. 
This design system is therefore based on the following principles: 
• Warm, intelligent, premium, and modern. 
• Calm background, sharp text, clear hierarchy. 
• Elegant motion, never chaotic motion. 
• Glassmorphism used for depth and emphasis, not decoration overload. 
• Cursor-reactive effects that feel alive, but never distract from parking operations. 
• Beige and brown as the base brand atmosphere. 
• Ginger as the active accent color. 
• High readability for every critical surface. 
• Strong consistency across dashboard, events, violations, rules, and settings.

3. Design Principles 
3.1 Warm-first interface 
The entire UI should avoid sterile white backgrounds and harsh monochrome contrast. 
Instead, it should use layered warm neutrals: 

===== PAGE 4 =====

• parchment beige 
• cream 
• oat 
• sand 
• almond 
• cocoa 
• dark brown 
This creates a more premium and differentiated experience. 
3.2 Glassmorphism with discipline 
Glassmorphism is part of ParkingSol’s identity, but it must be controlled. 
Glassmorphism depends primarily on opacity and background blur, and stronger blur is 
especially helpful on visually busy backgrounds because it preserves focus and 
readability. Therefore, ParkingSol should use frosted layers with enough blur to separate 
content from the background instead of using nearly transparent panels that make text 
harder to read.  
3.3 Cursor-reactive interface 
The product should feel alive when the cursor moves. However, motion should be subtle 
and should enrich the interface rather than turning every page into an animation demo. 
Allowed cursor-reactive effects: 
• soft radial highlight that follows cursor on hero or dashboard surfaces 
• micro-parallax on large decorative background orbs 
• hover bloom on cards 
• gentle gradient shift on buttons 
• magnetic hover on primary CTA buttons 
• row glow on interactive tables 
Disallowed cursor-reactive effects: 
• large lagging blobs that cover content 
• dramatic 3D rotation on operational tables 
• wobble, bounce, spring overload 
• forced cursor trails 
• effects that cause text blur or pointer distraction 
• flashy motion on every component at once 

===== PAGE 5 =====

3.4 Motion should support comprehension 
Motion must answer a question: 
• What changed? 
• What is interactive? 
• What just updated? 
• Where did this panel come from? 
• What state did this become? 
If motion does not serve one of those purposes, it should be removed. 
3.5 Accessibility is required 
Normal body text must meet a contrast ratio of at least 4.5:1, large text must meet at 
least 3:1, and UI components and visual indicators should meet at least 3:1 against 
adjacent colors. The interface must also respect prefers-reduced-motion, which is 
used to detect whether a user has enabled reduced motion settings on their device. 
This means: 
• beige can be beautiful, but not washed out 
• brown can be elegant, but not muddy 
• ginger accents can glow, but not obscure labels 
• motion can be expressive, but must quiet down when the user asks for less 
motion

9.4 Cursor-reactive rules 
ParkingSol should include subtle cursor response in these areas: 
• dashboard hero stat panel 
• glass cards 
• primary CTA buttons 
• navigation items 
• table rows 
• modal headers 
Approved cursor effects: 
• radial highlight follows cursor within large card 
• background orb shifts 2 to 8 px maximum 
• card border glow intensifies on hover 
• button highlight tracks mouse center 
• table row background softly brightens under hover 
Hard limits: 
• no component should rotate more than 1.5 degrees 
• no parallax layer should move more than 8 px 

===== PAGE 15 =====

• no cursor effect should delay pointer feedback 
• no essential data should move position in a way that harms readability 
9.5 Reduced motion policy 
If the user has reduced motion enabled, the interface must reduce or remove non-
essential motion because prefers-reduced-motion exists specifically to detect when 
users want motion minimized. In reduced-motion mode:  
• remove parallax 
• remove cursor-follow highlights 
• shorten transitions 
• replace moving entry animations with fade only 
• remove scale pop on hover 
• keep only essential state feedback

Specific rule

18. Accessibility Requirements 
ParkingSol’s warm aesthetic is allowed only if it remains accessible. 
Required rules 
• normal text minimum contrast: 4.5:1 
• large text minimum contrast: 3:1 
• UI component and graphic indicators minimum contrast: 3:1 
• reduced motion mode required via prefers-reduced-motion support  
• keyboard focus must always be visible 
• semantic labels required for icon-only controls 
• status must not be conveyed by color alone 
Design implication 
If a beige-brown combination looks beautiful but fails contrast, it must be adjusted. If a 
hover glow looks premium but hides focus state, it must be revised. If cursor motion 
feels alive but causes distraction, it must be reduced.

Implementation Notes 
Two constraints should stay visible while designing this interface: the product’s earlier 
architecture focused on a real-time workflow where plate events are validated in 
Firebase and unpaid vehicles trigger staff-facing review flows, so the UI must 
foreground live events, violations, and notifications rather than decorative marketing 
patterns. At the same time, glass surfaces and motion need guardrails, because 
readability improves when complex backgrounds are paired with stronger blur, normal 
text should reach at least 4.5:1 contrast, and motion should adapt when the user prefers 
reduced motion.
