1. Document Purpose 
This document defines the complete application flow for ParkingSol. Its purpose is to 
remove ambiguity from how users move through the product, what pages exist, what 
each page does, what actions are available on each page, and what the system should do 
after each action. This document is intentionally detailed so implementation can happen 
without guessing. 
ParkingSol is not a camera app. It is not a license plate recognition product. It is the 
parking operations software layer that receives parking events, applies rules, decides 
paid versus unpaid status, creates violations when necessary, updates the dashboard in 
real time, and notifies the correct users. 
This document answers five practical questions: 
1. What screens exist in the MVP? 
2. Who can access each screen? 
3. How does a user move from screen to screen? 
4. What happens after every important user action? 
5. What happens when the system receives an event even when no user is actively 
on the page? 
The app flow in this document assumes the following current product direction: 
• Frontend is a TypeScript web app. 
• Backend is Firebase Cloud Functions. 
• Database is Firestore. 
• Hosting is Vercel. 
• Authentication is Firebase Authentication. 
• Notifications are part of the MVP. 
• Testing of event ingestion starts with Postman. 
• UniFi is a future live event source, not required for initial build completion. 
• One Firebase project supports multiple authenticated users. 
• The product is an operator/admin dashboard, not a public customer-facing app. 
The earlier R&D already centered the system on receiving plate-related events, checking 
whether a plate has an active payment in Firebase, and alerting staff when a vehicle is 
unpaid, while analyzing all vehicles and preserving enforcement-focused records for 
violations. That earlier flow also assumed an app-facing review experience backed by

2. Application Flow Principles 
Before defining pages, the application needs consistent navigation rules. These 
principles apply everywhere in the product. 
2.1 Core flow principles 
• Every page must have a single primary purpose. 
• Every important action must have a visible next step. 
• No user should need to guess where to go after login. 
• The app should prioritize live operations, not static reporting. 
• The system should surface unpaid/exception events faster than compliant events. 
• Operators should reach active violations in one click from any core dashboard 
screen. 
• Admin tasks must be separated from live enforcement tasks to reduce accidental 
changes. 
• Destructive actions must require confirmation. 
• Permission restrictions must be visible and enforced. 
• The navigation must remain consistent across desktop layouts. 
2.2 Global layout principles 
After login, the app uses a persistent web dashboard shell with: 
• Left sidebar navigation. 
• Top bar with lot selector, notifications, search, user menu, and environment 
badge. 
• Main content area. 
• Optional right-side details drawer on selected pages. 
• Toast or banner system for success, warning, and error states. 
2.3 MVP navigation model 
Primary left navigation: 
• Dashboard 
• Live Events 
• Violations

• Vehicles 
• Notifications 
• Rules 
• Users 
• Settings 
• System Status 
The app should not hide key screens behind modal-only access. Core pages must have 
direct routes so users can bookmark or return to them. 
2.4 Role-aware navigation 
Different roles see different nav items: 
• Operator: Dashboard, Live Events, Violations, Vehicles, Notifications 
• Admin: All operator pages plus Rules, Users, Settings, System Status 
• Manager: Dashboard, Violations, Vehicles, Notifications, System Status 
• Support/Implementation: Dashboard, Live Events, Violations, Vehicles, 
Notifications, System Status, and test/integration visibility if enabled 
Pages that are unavailable to a user should either: 
• Not appear in navigation at all, or 
• Appear disabled only if the design intentionally communicates capability tiers. 
For MVP, hiding inaccessible pages is cleaner.

4.3 Global search behavior 
Search should accept: 
• Plate number. 
• Event ID. 
• Violation ID. 
• User name or email for admins. 
If a search matches:

The notification bell opens a panel or routes to the Notifications page. It must show: 
• Unread count

• Type badge 
• Timestamp

Unpaid vehicle alerts must be treated as high-visibility items because the earlier 
ParkingSol flow was built around detecting non-payment and surfacing alerts for staff 
review.

8. Cross-Page User Paths 
This section defines the exact journeys through the app. 
8.1 First-time admin login path 
1. Admin receives credentials or invite. 
2. Admin logs in at /login. 
3. Admin lands on /dashboard. 
4. Admin sees empty or near-empty system state. 
5. Admin goes to /settings to confirm lot info. 
6. Admin goes to /rules to configure core parking rules. 
7. Admin goes to /users to verify or add users. 
8. Admin goes to /system-status to verify environment is in test mode. 
9. Admin uses Postman externally to trigger test events. 
10. Admin returns to /events and /dashboard to validate incoming data. 
11. Admin opens /violations to confirm non-compliant flows work. 
12. Admin reviews /notifications to confirm alert delivery behavior. 
This is the initial implementation and setup journey. 
 
8.2 Operator daily flow 
1. Operator logs in. 
2. Operator lands on /dashboard. 
3. Operator checks summary cards and open violations. 
4. Operator clicks into /violations if there are unresolved items.

5. Operator opens /violations/:violationId. 
6. Operator reviews reason and related event. 
7. Operator moves to /vehicles/:plate if more context is needed. 
8. Operator acknowledges, resolves, or escalates the violation. 
9. Operator returns to /violations or /dashboard. 
10. Meanwhile, live updates continue to flow in. 
This should be the most frictionless path in the app. 
 
8.3 Support/testing flow with Postman

2. Support user opens /system-status. 
3. Confirms environment is test mode and backend is healthy.

7. If unpaid, opens /violations/:violationId. 
8. Opens /notifications to verify alert generation.

9. System-Driven Flows Without Manual Navigation 
Not every important app flow starts with a click. Some begin when the backend receives 
an event. 
9.1 Paid event system flow 
1. Event arrives through ingestion endpoint. 
2. Backend validates payload. 
3. Event is normalized. 
4. Plate is checked for active payment/authorization. 
5. Decision engine marks event compliant. 
6. Event record is stored. 
7. Dashboard counters and event stream update. 
8. Vehicle history updates. 
9. No violation is created. 
10. No urgent operator action is required. 

===== PAGE 27 =====

This follows the earlier system design in which the backend checks whether a plate has 
an active payment and records a lightweight event when compliant.  
 
9.2 Unpaid event system flow 
1. Event arrives through ingestion endpoint. 
2. Backend validates payload. 
3. Event is normalized. 
4. Plate lookup finds no valid active payment/authorization. 
5. Rule engine evaluates grace period and other exemptions. 
6. If non-compliant, a violation is created. 
7. Notification record is created. 
8. Dashboard updates. 
9. Violations page updates. 
10. Notifications page and bell update. 
11. Vehicle page now reflects open non-compliance state. 
This reflects the earlier operational concept in which unpaid vehicles trigger a violation-
style record and a staff-facing alert path.  
 
9.3 Duplicate event suppression flow 
1. Event arrives. 
2. Backend detects same plate and same event pattern within configured duplicate 
window. 
3. Event is recorded with duplicate-suppressed outcome or merged logic. 
4. Dashboard may show it in event history but should not create repeated noise. 
5. No duplicate violation should be created unless the system detects meaningful 
state change. 
 
9.4 Invalid payload flow 
1. Bad payload arrives. 
2. Validation fails. 
3. API returns error. 
4. Failure is logged. 
5. System status reflects processing issue if needed. 

===== PAGE 28 =====

6. Admin/support user can inspect failure context. 
 
9.5 Permission denial flow 
1. User attempts to access admin-only route. 
2. Route guard checks role. 
3. User is redirected to /unauthorized or back to dashboard. 
4. App does not leak restricted data.

13. Final Flow Contract 
ParkingSol’s application flow is built around one central operational truth: 
An event comes in, the system decides what it means, the right records 
update, and the right users can immediately see and act on the result. 

===== PAGE 30 =====

That means the MVP navigation must always support these actions with minimal 
friction: 
• Log in. 
• See current lot state. 
• Watch live events. 
• Identify unpaid vehicles. 
• Review and act on violations. 
• Search vehicle history. 
• Receive alerts. 
• Configure rules and users. 
• Verify system health. 
• Move from test-mode Postman events to future live-source readiness without 
changing the user-facing operating model. 
This is the required app flow for ParkingSol. If the implemented UI does not support 
these paths clearly, then the product is not complete, even if the backend logic exists.
