Board Setup 
Use this as the canonical issue board for ParkingSol MVP. The earlier R&D centered the 
system on webhook-style event intake, Firebase Cloud Functions validation, Firestore as 
the system of record, and unpaid alerting to staff, so this board is sequenced around that 
exact operational path rather than generic app-building tasks.

Execution Notes 
This issue board is intentionally ordered so that every later feature sits on a stable 
foundation instead of forcing rewrites. That matters for ParkingSol because the 
product’s value comes from a reliable chain of event intake, validation, Firestore-backed 
decisioning, and staff-facing enforcement visibility, not from isolated UI pages built 
before the backend can make trusted decisions.

Best order to open issues in GitHub 
Open all 45 issues at once, but assign them in this order: 
1. Issues 1–10 first 
2. Issues 11–18 second 
3. Issues 19–31 third 
4. Issues 32–36 fourth 
5. Issues 37–43 fifth 
6. Issues 44–45 last 
That order keeps the board realistic and reduces the chance that frontend work starts 
before backend contracts are stable. 
Recommended issue dependencies note 
At the bottom of each GitHub issue, add a short dependency note like: 
• Blocked by #6, #8 
• Unblocks #17, #29 
• Part of MVP critical path 
That will make the board much easier to manage during a 72-hour sprint.
