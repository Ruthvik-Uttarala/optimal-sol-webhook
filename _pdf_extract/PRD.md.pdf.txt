1. Product Overview 
Product name: ParkingSol 
Document purpose: 
This Product Requirements Document defines exactly what ParkingSol is building in the 
current product pivot, why it is being built, who it serves, what is included in the MVP, 
what is excluded, and what “done” means for an end-to-end working release. This 
document is intended to remove ambiguity before implementation starts so the 
engineering workflow can execute with minimal guessing and minimal rework. 
Current pivot summary: 
ParkingSol is not building a custom license plate recognition system. The camera 
system is responsible for detecting vehicles and reading plates. ParkingSol begins after 
plate data exists. The product being built is the operational software layer that receives 
parking events, applies parking rules, determines whether a vehicle is compliant or non-
compliant, creates violation records when required, updates a live dashboard, and 
notifies the correct users in real time. 
MVP statement: 
The first working MVP flow is: 
UniFi event simulation through Postman -> webhook/API ingestion -> 
Firebase Cloud Functions -> Firestore -> paid/unpaid decision -> live 
dashboard update -> notification to the right user 
During development, UniFi will be replaced by Postman so the full system can be built, 
tested, debugged, and validated before the UniFi console is purchased and connected. 
Primary objective: 
Deliver a real, deployable, end-to-end working ParkingSol MVP in 72 hours, with all 
core logic functioning, all required APIs connected, all code committed to GitHub, and 
the system ready for later live integration with a UniFi environment. 
Core product modules: 
ParkingSol MVP includes the following major modules: 
• Rules engine. 
• Parking intelligence and decisioning. 
• Violations system.

2. Problem Statement 
Parking operations break when there is no reliable software layer between plate capture 
and enforcement action. Cameras may detect the plate, but without a structured 
backend, operators still face slow manual checks, inconsistent rule application, poor 
visibility into unpaid vehicles, fragmented records, and no dependable alerting. 
The deeper problem is not just automation. It is decision consistency. A parking 
operation needs to answer the same questions every time, with clear logic: 
• Did this vehicle enter? 
• Is there a valid paid session or permit? 
• Does the vehicle belong to a resident, guest, staff member, or visitor? 
• Is the vehicle still inside the grace period? 
• Is this an entry, exit, repeat read, or duplicate event? 
• Should staff be notified now? 
• Should a violation be created now, later, or not at all? 
• What evidence and audit trail must be preserved? 
Without a dedicated rules engine and operations dashboard, every one of these 
decisions becomes inconsistent, manual, or hidden inside ad hoc scripts. That makes the 
system unreliable, hard to debug, and impossible to scale to multiple users or future 
lots. 
ParkingSol solves this by becoming the central parking intelligence layer. It accepts 
parking events, normalizes them, applies deterministic rules, writes the result to 
Firestore, displays current state in a live dashboard, and triggers notifications when 
action is required. It creates a repeatable operating model instead of a loose collection of 
integrations.

3. Product Vision and Goals 
Vision 
ParkingSol will be the operating system for parking enforcement and parking visibility 
at small to mid-sized lots that already have, or will soon have, plate capture hardware. It 
will provide a shared backend for multiple users, real-time operational awareness, a 
clean rules engine, and a practical workflow for identifying and acting on non-compliant 
vehicles. 
Mission 
Convert raw plate events into trusted parking decisions with minimal delay, minimal 
manual review, and complete traceability. 
Product goals 
1. Build the software layer, not the camera layer. 
ParkingSol will rely on external plate capture systems such as UniFi rather than 
building custom OCR or camera inference logic. 
2. Achieve a working end-to-end MVP in 72 hours. 
The MVP must be fully testable with Postman and deployable before live UniFi 
integration. 
3. Use a TypeScript-first stack. 
Frontend and backend implementation should be consistent, maintainable, and 
easy to hand off to AI coding workflows and human developers. 
4. Use Firebase as the operational backend. 
Cloud Functions, Firestore, Authentication, and notifications must serve as the 
core system services. 
5. Host the frontend on Vercel. 
The operator dashboard and admin web application must be accessible via a 
production URL and be ready for real use. 
6. Support multiple authenticated users in one shared system. 
A single Firebase project will power the MVP, but users must be logically 
separated using authentication, roles, and controlled data access. 
7. Provide real-time operational visibility. 
When events arrive, the dashboard must update live and relevant users must see 
the resulting status quickly.

8. Preserve auditability. 
Every important decision should leave a record that explains what happened, why 
it happened, and when it happened. 
Business goals 
• Reduce manual parking validation work. 
• Reduce delayed violation response. 
• Provide a clean path from simulated testing to live hardware integration. 
• Create a stable foundation for future production deployment across additional 
lots or customers. 
• Make the system specific enough that coding tools can implement it without 
inventing architecture. 
Product principles 
• Documentation first. 
• Deterministic rules over vague heuristics. 
• Minimal hidden logic. 
• Real-time by default. 
• Multi-user from day one. 
• Clear separation between in-scope MVP and future enhancements. 
• Production-minded even when testing with mocks. 
 
4. Users and Personas 
ParkingSol MVP is designed primarily for operational and administrative users rather 
than public end users. 
Primary personas 
1. Parking operator 
This user monitors incoming events, reviews unpaid vehicles, checks whether alerts are 
valid, and uses the dashboard to understand current lot activity. They need speed, 
clarity, and low cognitive load. They should be able to tell within seconds whether a 
vehicle requires attention. 
2. Parking administrator 
This user configures the lot, rules, payment validation behavior, user access, notification

settings, and enforcement defaults. They need control, visibility, and confidence that 
rule changes will apply predictably. 
3. Lot owner or manager 
This user wants a clear picture of occupancy patterns, compliance, violations, and 
system health. They do not need deep technical controls, but they need confidence that 
the lot is being managed correctly. 
4. Support or implementation user 
This user tests the system with Postman, validates API behavior, reviews logs, and 
confirms that event ingestion and decisioning work correctly before live UniFi 
integration. They need observability and repeatable test flows. 
Secondary personas 
5. Enforcement reviewer 
This user may not monitor every event live but needs a reliable list of unresolved or 
recently flagged violations. 
6. Future client admin for a new lot 
Not fully in MVP, but the system should be designed so the shared backend can later 
support additional customer accounts without total rework. 
User needs 
Across personas, the shared user needs are: 
• Know whether a vehicle is compliant. 
• Understand why a decision was made. 
• Be alerted when action is needed. 
• Avoid duplicate or noisy alerts. 
• Filter events by time, plate, status, or lot area. 
• Trust that their data is secure and role-appropriate. 
• Use the product in real time without page refreshes or manual polling.

Core functional pillars 
A. Rules engine 
This is the heart of the system. The rules engine decides what a parking event means 
and what the platform should do next. It will evaluate events using plate number, 
timestamp, lot configuration, vehicle state, permit/payment status, grace windows, and 
event type. 
Example decisions include: 
• Paid and compliant. 
• Unpaid and immediately violative. 
• Unpaid but still inside grace period. 
• Duplicate read that should not create another alert. 
• Exit event that should close an active parking state. 
• Unknown plate requiring review. 
• Whitelisted plate with no violation. 
• Reserved vehicle with custom rule behavior. 
B. Parking intelligence 
ParkingSol must maintain a structured understanding of active vehicle state rather than 
treating each event in isolation. For example, the platform should know whether a

vehicle is believed to be currently inside the lot, whether it has an active session, 
whether it has an unresolved violation, and whether a new event changes that state. 
C. Violations system 
When a vehicle is non-compliant based on current rules, the system must create a 
violation record with status, reason, timestamps, linked event data, and reviewability. 
The violation system is not just a flag. It is a trackable operational object. 
D. Dashboard 
The dashboard must show what operators need now, not just raw data. This includes 
recent events, current unpaid vehicles, recent alerts, active violations, lot status 
summary, and searchable vehicle history. 
E. Notifications 
Relevant users must be informed when key events occur. At MVP stage, notifications 
may include in-app notifications and other practical notification channels that can be 
implemented reliably in the 72-hour window. The critical requirement is that important 
unpaid or exception events are surfaced immediately to the right user. 
 
6. Detailed Feature Requirements 
6.1 Event ingestion 
The system must expose secure endpoints that accept parking event payloads. For MVP, 
these payloads come from Postman. Later, the same ingestion layer should be adaptable 
to UniFi webhook payloads with minimal translation work. 
Each event payload should support, at minimum: 
• External event ID. 
• Event source. 
• Plate text. 
• Timestamp. 
• Camera or source identifier. 
• Direction if known, such as entry or exit. 
• Confidence or metadata if available. 
• Snapshot URL or placeholder evidence metadata if available. 
• Raw payload storage for debugging. 

===== PAGE 8 =====

The system must validate payload structure and reject malformed events with clear 
error responses. 
6.2 Event normalization 
Incoming events may vary by source over time. ParkingSol must normalize them into a 
single internal event shape before business rules are applied. This ensures that Postman 
simulation today and UniFi ingestion later feed the same downstream logic. 
6.3 Decision engine 
The decision engine must evaluate each normalized event against current parking data 
and lot rules. It should determine: 
• Whether the plate has a valid active payment or permit. 
• Whether the event is entry, exit, unknown, or duplicate. 
• Whether the event should create, update, or close an active parking session. 
• Whether a violation should be created. 
• Whether a notification should be sent. 
• Whether the event should be marked as reviewed or pending. 
6.4 Session and state tracking 
The system must maintain active parking state for vehicles. Even if a vehicle produces 
multiple events, the backend should preserve a coherent state rather than treating each 
read as unrelated. 
Examples: 
• Entry event opens or updates a presence record. 
• Exit event closes an active presence record. 
• Duplicate events within a short time window do not create noise. 
• A plate with a valid permit may remain compliant throughout its stay. 
• A plate that transitions from unpaid to paid must be reflected correctly. 
6.5 Violations workflow 
When the rules engine determines a violation exists, the system must: 
• Create a violation record. 
• Attach the triggering event. 
• Store status such as open, acknowledged, dismissed, resolved, or escalated. 
• Preserve reason codes.

• Display the violation immediately in the dashboard.

Business logic rules for MVP 
The MVP must support the following rule classes: 
1. Paid session validation 
A plate is compliant if a valid active payment or authorization exists for the 
evaluated time window. 
2. Permit or whitelist validation 
A plate may be exempt due to admin-defined authorization such as resident, 
staff, vendor, or allowlist status. 
3. Grace period evaluation 
A vehicle may have a configurable grace period after entry before it becomes 
violative. 
4. Duplicate suppression 
Multiple reads of the same plate in a short window should not create duplicate 
actions unless meaningful state changed. 
5. Direction-aware handling 
If event direction is known, the engine should use it to update presence and close 
loops between entry and exit. 
6. Violation thresholding 
Rules may define when a non-compliant read becomes a formal violation versus a 
warning state. 
7. Notification routing 
Different rule outcomes may notify different user roles. 
Rule configurability 
Rules must be configurable enough to support real parking operations, but not so 
abstract that the MVP becomes a generic workflow engine. This is a parking rules 
system, not a no-code platform. 
The first version should allow administrators to define or edit: 
• Lot-level grace period. 
• Active enforcement hours. 
• Plate allowlists. 
• Permit types. 
• Notification recipients by event category. 
• Duplicate suppression window. 
• Default violation severity or review status.

10. End-to-End MVP Workflow 
The MVP operational workflow is: 
1. A test event is sent from Postman to the ParkingSol ingestion endpoint. 
2. The backend authenticates or validates the request. 
3. The payload is normalized into ParkingSol’s internal event format. 
4. The event is stored for audit and traceability. 
5. The decision engine checks active payments, permits, allowlists, and rule 
configuration. 
6. The engine determines the vehicle status. 
7. If compliant, the system records the event and updates dashboard state. 
8. If non-compliant, the system creates a violation object and triggers notifications. 
9. Firestore updates propagate to the dashboard in real time. 
10. Authenticated users see the updated event feed, violation list, and vehicle history. 
11. Operators or admins can review, acknowledge, dismiss, or resolve records 
according to permissions. 
12. All code and configuration remain structured so the simulated source can later be 
replaced with a UniFi webhook source. 
This workflow is the MVP contract. Any implementation that does not complete this 
chain is incomplete.

13. Risks, Constraints, and Assumptions 
Risks 
• Ambiguous scope can cause AI-generated implementation drift. 
• Real UniFi payload shape may differ from the mock payload if not normalized 
carefully. 
• Notification complexity can expand quickly if too many channels are attempted. 
• Multi-user permissions can become error-prone without early discipline. 
• Time pressure can push logging, validation, and error handling aside unless 
explicitly required. 
Constraints 
• Delivery target is 72 hours. 
• Frontend must be TypeScript and not FlutterFlow. 
• Backend must use Firebase services. 
• Database must use Firestore. 
• Frontend hosting must use Vercel. 
• UniFi hardware integration is deferred; Postman is the current test substitute. 
• AWS is not part of MVP. 
Assumptions 
• Plate text is available from the upstream source or simulated payload. 
• One Firebase project can support the initial user base. 
• Initial lot configuration can be represented in Firestore. 

===== PAGE 17 =====

• Payment or permit status can be modeled in Firestore for MVP decisioning. 
• The first release is operations-facing, not customer-facing.

15. Final Product Statement 
ParkingSol MVP is a real-time parking operations platform that starts after plate 
capture, ingests parking events, applies configurable parking rules, determines paid 
versus unpaid status, creates violations when needed, updates a live dashboard, and 
notifies authorized users. It is built as a TypeScript web application on Vercel with 
Firebase Cloud Functions, Firestore, and Authentication, tested first through Postman, 
structured for a shared multi-user backend, and designed to be ready for later live UniFi 
webhook integration. 
That is the product being built. Anything outside that statement is either a future phase
