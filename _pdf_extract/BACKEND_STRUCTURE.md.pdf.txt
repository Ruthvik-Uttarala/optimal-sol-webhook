1. Document Purpose 
This document defines the complete backend structure for ParkingSol. It is the 
canonical source of truth for the backend architecture, Firestore schema, collection 
design, subcollections, document fields, field types, relationships, API endpoints, 
authentication behavior, authorization behavior, processing flow, audit requirements, 
and system rules needed to build the MVP end to end. 
This document exists to eliminate backend ambiguity. It is intentionally specific so 
implementation can happen without guessing. The goal is not to create a theoretical 
architecture. The goal is to define a backend that can be built quickly, deployed cleanly, 
tested through Postman, connected to the frontend, and later switched from mock event 
intake to live UniFi webhook intake with minimal redesign. 
ParkingSol is not building custom LPR or camera vision. The camera or upstream 
source is responsible for detecting the vehicle and providing a plate read. ParkingSol 
begins once an event payload exists. The backend must then do the following: 
• Receive an event. 
• Validate it. 
• Normalize it. 
• Store it. 
• Determine whether the vehicle is paid, unpaid, exempt, or pending review. 
• Update vehicle state. 
• Create a violation when required. 
• Notify the correct users. 
• Make all resulting data available to the web dashboard in real time. 
The earlier R&D already described this same operational chain in practical terms: an 
event comes in through a webhook, Cloud Functions validates it, Firestore is queried to 
check for an active paid session, lightweight event records are stored for compliant 
cases, and violation records plus notifications are created for non-compliant cases. The 
same earlier work also treated Firestore as the system of record for paid plate sessions, 
event metadata, and violation records, which directly informs the collection structure 
defined in this backend document.  

===== PAGE 2 =====

This document treats “table” to mean the Firestore equivalent of a collection or a 
strongly defined subcollection. Because Firestore is not a relational SQL database, 
ParkingSol will use collections, documents, and subcollections rather than SQL tables 
and joins. Even so, every major entity is defined as rigorously as if this were a relational 
schema. 
 
2. Backend Mission 
The ParkingSol backend has one job: 
Turn raw parking event inputs into trusted operational records and actions. 
That requires five backend responsibilities: 
1. Ingest event payloads safely. 
2. Run deterministic rule evaluation. 
3. Maintain current vehicle and lot state. 
4. Create operational objects such as violations and notifications. 
5. Serve a secure API to the frontend and support tools. 
The backend is not a generic CMS. It is not a data dump. It is the system that decides 
what each parking event means. 
2.1 Core backend principles 
• Firestore is the source of truth. 
• All write paths go through backend validation. 
• No critical business rule runs only in the frontend. 
• Every major action is auditable. 
• Duplicate events must not create duplicate violations. 
• Multi-user architecture is mandatory from day one. 
• Postman-based event testing must work before UniFi integration. 
• Schema names must match product language exactly. 
• Status fields must be explicit, never implied. 
• Timestamps must be written server-side wherever possible.

3. Backend Architecture Overview 
ParkingSol backend follows this architecture: 
Postman or future UniFi source -> Cloud Functions HTTP endpoint -> 
validation -> normalization -> Firestore lookup -> decision engine -> 
Firestore writes -> notifications -> frontend real-time updates 
This matches the operational pattern already established in the earlier research, where 
Cloud Functions received webhooks, queried Firestore for payment state, wrote event 
and violation records, and surfaced results to staff-facing application workflows.

3.2 Layer responsibilities 
routes 
Maps HTTP routes to controllers. 
controllers 
Handles request parsing, response formatting, and controller-level error boundaries. 
services 
Contains business logic such as event processing, payment validation, rule evaluation, 
violation creation, and notification logic. 
repositories 
Contains Firestore read/write functions only. No business logic. 
schemas 
Contains Zod validation schemas for request bodies, params, queries, and internal 
normalized payloads. 
middleware 
Contains auth verification, role checking, request ID injection, environment gating, and 
error handling middleware. 
types 
Contains shared domain types for entities such as event, violation, rule, vehicle state, 
payment, permit, and notification. 

===== PAGE 5 =====

utils 
Contains generic reusable helpers such as plate normalization, date calculations, 
duplicate key generation, pagination parsing, and response builders. 
config 
Contains environment configuration, collection names, route constants, status enums, 
and operational defaults. 
3.3 Function grouping strategy 
ParkingSol should expose one HTTP API app under a Cloud Functions deployment 
namespace, rather than many disconnected single-function routes. The backend should 
feel like one coherent API. 
Recommended root: 
• /api/v1/* 
Webhook-style ingestion routes can still live under: 
• /api/v1/webhooks/*

4.2 Relationship philosophy 
ParkingSol uses Firestore document references and duplicated summary fields where 
needed for operational performance. It does not try to simulate heavy SQL joins on 
every request. 
Example: 
• A violation stores plate, lotId, eventId, and vehicleStateId directly. 
• A notification stores targetUserId, type, linkedViolationId, and a 
lightweight message summary. 
• An event stores normalizedPlate, decisionSummary, and linkedViolationId 
if one exists. 
This is intentional. ParkingSol is an event-driven operational system, so read speed and 
traceability matter more than strict normalization purity.

6. Firestore Index Strategy 
ParkingSol should define composite indexes intentionally. 
Required index patterns 
• events by lotId + capturedAt desc 
• events by normalizedPlate + capturedAt desc 
• events by processingStatus + capturedAt desc 
• events by decisionStatus + capturedAt desc 
• violations by lotId + status + createdAt desc 
• violations by normalizedPlate + createdAt desc 
• notifications by targetUserId + isRead + createdAt desc 
• payments by lotId + normalizedPlate + validUntil desc 
• permits by lotId + normalizedPlate + status 
• rules by lotId + status + priority 
• parkingSessions by lotId + normalizedPlate + openedAt desc 
Notes 
The backend must not rely on ad hoc Firestore console prompts during development. 
Required indexes should be defined and committed.

10. Processing Flow 
10.1 Event processing sequence 
1. Receive request. 
2. Validate API client or user auth. 
3. Validate payload schema. 
4. Resolve source and lot context. 

===== PAGE 34 =====

5. Normalize plate and event shape. 
6. Generate dedupe key. 
7. Check processing lock. 
8. Create initial event record with processingStatus = received. 
9. Evaluate active payment. 
10. Evaluate active permit. 
11. Load applicable rules. 
12. Determine duplicate suppression. 
13. Update or create vehicle state. 
14. Open, update, or close parking session as needed. 
15. Determine final decision status. 
16. If needed, create violation. 
17. If needed, create notifications. 
18. Mark event processed. 
19. Write audit logs. 
20. Return response. 
This follows the general operational pattern already defined earlier, where Cloud 
Functions validates inbound data, checks Firestore for authorization state, writes the 
event result, and creates violation and notification records for unpaid cases.  
10.2 Decision order 
The decision engine should evaluate in this order: 
1. Payload validity. 
2. Source validity. 
3. Duplicate suppression. 
4. Permit/allowlist validity. 
5. Active payment validity. 
6. Grace period rules. 
7. Enforcement hours rules. 
8. Manual review rules. 
9. Violation creation rules. 
10. Notification routing rules. 
10.3 Paid outcome 
If payment or permit is valid: 
• event marked paid or exempt 
• vehicle state updated 

===== PAGE 35 =====

• session updated 
• no violation created 
• optional low-priority notification none by default 
This maps to the earlier flow where compliant reads produce lightweight event records 
without violation storage.  
10.4 Unpaid outcome 
If no payment or permit exists and no grace exemption applies: 
• event marked unpaid 
• vehicle state updated 
• violation created if threshold met 
• notification created 
• session linked to violation if relevant 
This matches the earlier flow where unpaid reads led to violation records and staff 
alerts.

18. Final Backend Contract 
ParkingSol’s backend is a Firebase-based event decision system. It receives parking 
events, validates and normalizes them, checks active payment or permit state, applies 

===== PAGE 41 =====

configurable rules, updates current vehicle state, maintains session records, creates 
violations when required, generates notifications, and exposes secure APIs for 
operators, admins, managers, and support users. 
That is the backend being built. The collections above are the required schema. The 
routes above are the required API surface. The processing flow above is the required 
business logic path. If the implementation omits any of those critical structures, then 
the backend is not actually complete, even if parts of the app appear to work. 
Notes 
The reason this backend is structured around payments, events, parkingSessions, 
violations, and notifications is that the earlier ParkingSol R&D already defined the 
operational model as plate-event intake, Firebase-based validation against active 
payment state, lightweight recording of compliant events, and violation-plus-alert 
creation for unpaid cases. That same earlier architecture also treated Firestore as the 
record system for paid sessions and violation records, which is why the schema here is 
centered on explicit Firestore collections instead of external SQL or AWS-backed 
storage for MVP.
