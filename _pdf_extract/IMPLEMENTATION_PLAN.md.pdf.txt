1. Document Purpose 
This document defines the exact implementation sequence for ParkingSol MVP. It is the 
build order for the real product, not a loose task list. Its purpose is to ensure the project 
can be completed end to end in a tightly controlled 72-hour sprint without architectural 
drift, duplicated work, or AI-generated guesswork. 
This plan is designed around the current confirmed product direction: 
• ParkingSol is not building custom LPR. 
• ParkingSol begins after a plate event already exists. 
• MVP flow is Postman -> webhook/API -> Firebase Cloud Functions -> 
Firestore -> rules/decision engine -> violation/notification -> live 
dashboard. 
• Frontend is a TypeScript web app. 
• Backend is Firebase Cloud Functions in TypeScript. 
• Database is Firestore. 
• Frontend hosting is Vercel. 
• One Firebase project supports multiple authenticated users. 
• Notifications are included in MVP. 
• AWS is removed from MVP. 
• Code must be pushed to GitHub. 
• Product must be ready for later UniFi integration without re-architecting the 
system. 
The earlier R&D established the same core sequence in practical terms: detect plate 
upstream, send event metadata through a webhook, validate in Firebase, check Firestore 
for active payment, and create an unpaid alert workflow when required. Firebase’s 
current Cloud Functions support for Node 20 and the exact pinned stack already chosen 
for the frontend and backend also make this plan realistic as a unified TypeScript 
implementation rather than a mixed-tool experiment.

2. Implementation Philosophy 
ParkingSol must be built in vertical slices, not isolated layers. That means each major 
implementation phase should aim to produce something that can actually be exercised, 
not just a pile of unconnected code. 
For example: 
• Do not build the whole frontend without a live backend. 
• Do not build the whole backend without a route that can be hit from Postman. 
• Do not build rules logic before payload validation and schema shape are stable. 
• Do not build polished UI before route structure, auth shell, and core data views 
exist. 
2.1 The sequence rule 
Every implementation step must answer one of these questions: 
• Does it create the foundation for everything else? 
• Does it enable end-to-end testing? 
• Does it remove a major integration risk? 
• Does it unlock the next phase cleanly? 
If the answer is no, that step should not be prioritized early.

2.2 Delivery objective 
At the end of 72 hours, ParkingSol should be in this state: 
• Repo exists in GitHub and all code is pushed. 
• Frontend is deployed on Vercel. 
• Backend is deployed on Firebase. 
• Authentication works. 
• Postman can send a parking event into the live backend. 
• Backend makes a paid/unpaid/exempt decision. 
• Firestore stores all required records. 
• Violations are created for non-compliant cases. 
• Notifications are visible in the app. 
• Dashboard updates in real time. 
• Admin/operator flows work. 
• The system is cleanly structured for future UniFi webhook connection. 
That is the definition of success for this sprint. 
2.3 Build constraints 
The build must optimize for: 
• speed 
• reliability 
• clarity 
• low ambiguity 
• low dependency risk 
• easy debugging 
• deployability 
The build must not optimize for: 
• speculative features 
• over-engineering 
• broad future platform abstractions 
• premature analytics 
• highly customized infrastructure 
• pixel-perfect polish before the core workflow works

3. Sprint Structure 
The 72-hour build will be split into seven phases: 
1. Foundation and repository setup 
2. Firebase and backend skeleton 
3. Firestore schema and seedability 
4. Event ingestion and rules pipeline 
5. Frontend shell and authenticated app 
6. Core operational pages and live data 
7. QA, hardening, deployment, and final push 
Each phase contains step numbers that must be implemented in sequence. 
3.1 Phase order summary 
• Phase 0: Lock environment and repo. 
• Phase 1: Create backend framework and auth skeleton. 
• Phase 2: Create Firestore schema support and test data paths. 
• Phase 3: Build event ingestion and decision engine. 
• Phase 4: Build frontend app shell and navigation. 
• Phase 5: Connect UI to live backend and Firestore. 
• Phase 6: Test all critical flows and deploy. 
• Phase 7: Final readiness review for future UniFi integration. 
3.2 Parallel work rule 
Some work can happen in parallel, but only after the dependencies are established. 
Allowed parallel examples: 
• frontend layout can begin once route map and data contracts are stable 
• UI component creation can happen while backend endpoints are being finalized 
• QA scripts can be drafted while dashboard pages are being connected 
Not allowed: 
• finalizing frontend data tables before API response contracts exist 
• building notifications UI before notification schema exists 
• building system status page before backend health endpoint exists 
• styling details before login and dashboard routes function

13. What Not to Build During This Sprint 
To protect the implementation sequence, do not divert time into: 
• custom LPR or OCR work 
• mobile app work 
• SMS/email complexity unless absolutely necessary 
• deep analytics dashboards 
• advanced charting 
• evidence image pipelines unless needed immediately 
• public driver-facing flows 
• invoice/billing systems 
• legal citation workflows 
• map views 
• enterprise multi-tenant onboarding UX 
• camera configuration UI 
• custom design experiments beyond the locked system 
• performance micro-optimizations before correctness is proven

The earlier R&D focused the real operational value on plate-event handling, Firebase-
backed validation, and unpaid alerting, so the implementation plan should stay centered 
on that path instead of expanding into adjacent systems.  
 
14. Responsibility Model for AI-Assisted Build 
Because this project is intended to be executed with heavy AI coding assistance, tasks 
should be framed in implementation-safe units. 
Good AI task units 
• create Zod schema for Postman event payload 
• build POST /api/v1/webhooks/postman/events 
• create Firestore repository for violations 
• build dashboard summary cards from /system/metrics 
• build notifications list page with unread filter 
• add violation acknowledge endpoint and UI button 
Bad AI task units 
• build the whole backend 
• make the dashboard amazing 
• connect everything automatically 
• improve architecture as needed 
• add anything missing 
The smaller and clearer the task, the less the AI invents. 
Human review checkpoints 
A human should review after: 
• repo setup 
• auth setup 
• schema definition 
• ingestion endpoint 
• rule engine 
• violation logic 
• first dashboard connection 
• production deploy

15. Completion Standard 
ParkingSol MVP is complete when the actual production-deployed system behaves like 
the documented product, app flow, tech stack, frontend system, and backend structure. 
It is not complete when files merely exist. It is complete when the product works end to 
end. 
The most important completion proof is this: 
1. Open the deployed ParkingSol web app. 
2. Log in as an authorized user. 
3. Send a test event from Postman to the deployed ingestion endpoint. 
4. Watch the backend process it. 
5. Confirm Firestore updates. 
6. Confirm the dashboard updates live. 
7. Confirm unpaid behavior creates a violation and notification. 
8. Confirm the operator can act on that violation in the UI. 
9. Confirm all code is in GitHub. 
If that sequence works, the MVP is real. If it does not, the implementation is not 
finished. 
Notes 
This implementation plan is intentionally ordered around the same operational chain 
already identified in the earlier ParkingSol research: incoming plate event, Firebase 
validation, Firestore state update, unpaid detection, and staff-facing alert flow. It also 
stays aligned with the exact stack already locked for the project, which keeps the build 
on a single Node 20-compatible Firebase backend and a pinned React 18.2.0 plus Vite 
frontend rather than introducing version drift or framework ambiguity during the 72-
hour sprint.
